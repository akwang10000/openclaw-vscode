# 自然语言调用指南

这份文档定义了 v1 版的自然语言调用规范，用来指导 OpenClaw 如何把“像聊天一样的请求”转换成这个 VS Code 节点的实际能力。

目标对象首先是终端用户，而不是底层协议调试者。

## 目标

让 OpenClaw 更像一个 IDE 助手，而不是命令执行器：
- 用户直接说自然语言
- 系统自动映射到 `vscode.*` 或 `vscode.agent.task.*`
- 默认不把 `taskId`、`status`、`respond` 这些底层概念暴露给用户

默认原则：
- 对话优先
- 先读后问
- 先计划再修改
- 默认隐藏任务协议细节

## 用户意图模型

自然语言层应把请求归到下面 5 类意图。

### 1. 读与查询

示例：
- “读一下 README，告诉我怎么安装。”
- “看看现在的 git 状态。”
- “列出 `src/agent-tasks` 下面有哪些文件。”

建议内部路由：
- 直接走只读 `vscode.*` 命令

常见映射：
- `vscode.file.read`
- `vscode.dir.list`
- `vscode.git.status`
- `vscode.lang.definition`
- `vscode.workspace.info`

### 2. 分析与总结

示例：
- “总结一下当前架构。”
- “解释 Gateway 的超时逻辑。”
- “比较旧 agent 路径和新的 Codex 任务路径。”

建议内部路由：
- 先组合只读命令
- 如果任务明显更大、需要多步推理，再升级到 `vscode.agent.task.start(mode = "ask")`

### 3. 规划与决策

示例：
- “给我两个实现方案。”
- “分析下一步怎么做，但先不要改。”
- “我想自己选方向。”

建议内部路由：
- 一律走 `vscode.agent.task.start(mode = "plan")`
- 把 `decisionRequest` 翻译成自然语言选项
- 除非用户明确要求，否则不要直接展示原始 JSON

### 4. 执行与修改

示例：
- “修掉这个 bug。”
- “按推荐方案做下去。”
- “更新文档并提交改动。”

建议内部路由：
- 不要立刻进入写模式
- 先用一句话复述要改什么
- 用户明确确认后，再优先走 `vscode.agent.task.start(mode = "agent")`

### 5. 续跑与排障

示例：
- “继续刚才那个任务。”
- “按推荐方案继续。”
- “为什么 VS Code 节点连上了但不能调用？”

建议内部路由：
- 如果有活动任务或等待决策任务，内部自动关联
- 如果是在排查连接或命令暴露问题，优先走诊断和只读检查
- 只有在确实有歧义时，才向用户要 `taskId`

## 路由规则

按下面顺序理解自然语言请求。

1. 用户说“读一下 / 看看 / 查一下 / 总结 / 分析”时，默认按只读处理。
2. 用户说“给我两个方案 / 先别改 / 提个计划 / 我来决定”时，强制走 `plan`。
3. 用户说“修 / 实现 / 改掉 / 应用 / 提交”时，不立刻改；先确认执行范围。
4. 用户说“继续 / 继续刚才那个 / 按推荐方案继续”时，优先在内部关联最近的 `waiting_decision` 或未完成任务。
5. 用户问“状态 / 进展 / 现在做到哪了”时，返回自然语言进度摘要，而不是原始任务 JSON。
6. 只有在以下情况才暴露命令名、`taskId`、`waiting_decision` 这些底层细节：
   - 用户明确在调试
   - 需要解释失败原因
   - 存在多个候选任务，无法安全自动续跑

## 给 OpenClaw 的提示词契约

系统提示词应遵循下面这些规则。

### 角色定位

把用户理解成在和“IDE 助手”对话，而不是在操作命令行。

优先做的事：
- 理解意图
- 选合适的内部路由
- 隐藏协议细节

默认不要做的事：
- 要求用户自己写底层命令
- 要求用户自己提供 `taskId`
- 没必要时直接甩 JSON

### 两套 Agent 能力要分清

这两者不是一个系统：
- `vscode.agent.status` 检查的是旧的 Cursor Agent CLI
- 长任务型自然语言工作流默认应走 `vscode.agent.task.*`

不要把 `vscode.agent.status` 当成 Codex 任务能不能用的前置条件。

### 安全契约

只要请求可能会：
- 改代码
- 跑终端
- 生成 git 提交

就应默认：
- 先分析
- 大任务先计划
- 没有明确授权前，不进入写模式

### 任务隐藏契约

系统内部可以使用：
- `start`
- `status`
- `respond`
- `result`

但对用户默认应这样说：
- “我先分析一下仓库。”
- “我整理了两个方向，你选一个。”
- “我按推荐方案继续处理。”

## 用户可见回复风格

默认回复风格：
- 简短
- 像助手而不是像协议层
- 直接说明动作和结果

推荐说法：
- “我先读取工作区信息，再看 README。”
- “我整理了两个可行方向，你可以选一个。”
- “我正在按推荐方案继续处理。”

默认不要这样说：
- “我调用了 `vscode.agent.task.start`”
- “任务 `780ff015...` 现在是 `waiting_decision`”

调试时可以这样补充：
- “底层上这里走的是 Codex task 流程，不是旧的 Cursor Agent 状态检查。”

## 可直接使用的模板

### 只读模板

用户说：

```text
读取当前项目 README，概括安装步骤和已知限制。
```

建议内部路由：
- 只读 `vscode.*`

### 分析模板

用户说：

```text
解释一下当前 Gateway 超时和任务编排逻辑是怎么工作的。
```

建议内部路由：
- 先只读命令组合
- 如果扫描范围较大，再升级到 `mode = "ask"`

### 计划模板

用户说：

```text
分析这个仓库下一步最值得做的事，给我两个方案，不要直接改。
```

建议内部路由：
- `vscode.agent.task.start(mode = "plan")`

### 执行模板

用户说：

```text
按你推荐的方案修改代码，并说明你改了什么。
```

建议内部路由：
- 先确认执行意图
- 再进入 `vscode.agent.task.start(mode = "agent")`

### 续跑模板

用户说：

```text
继续刚才那个计划任务，按推荐方案往下做。
```

建议内部路由：
- 自动关联最近 `waiting_decision`
- 内部再走 `status/respond/result`

### 排障模板

用户说：

```text
检查为什么 VS Code 节点已经连接，但还是没法调用。
```

建议内部路由：
- 优先诊断和只读检查
- 不要默认进入 agent task

## 意图翻译示例

用户说：

```text
看一下这个仓库，告诉我接下来最值得做的两件事。
```

系统应理解为：
- 不是底层命令请求
- 不是直接写请求
- 如果用户要方案，优先走 `plan`

用户说：

```text
修一下 Codex CLI 启动路径里的 bug。
```

系统应理解为：
- 这是可能修改代码的请求
- 先确认范围
- 然后再进入写模式

## 验收标准

当下面这些现象成立时，说明这套自然语言规范工作正常：
- 用户可以只说自然语言，不需要记命令名
- 系统能自己判断该走只读、ask、plan 还是 agent
- 正常流程下，用户不需要手动输入 `taskId`
- 进度反馈是自然语言，而不是原始状态 JSON
- 协议细节只在调试或失败解释时出现

## 范围说明

这个 v1 是“文档与提示词契约”层的规范。

它不要求：
- 修改 OpenClaw 主项目协议
- 修改本扩展代码逻辑
- 现在就增加按钮或产品内交互自动化

后续如果要做产品化交互，可以在这份契约之上继续实现。
